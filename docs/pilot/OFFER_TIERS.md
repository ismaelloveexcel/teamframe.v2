# TeamFrame Pilot Offer Tiers

Rule: only two tiers are allowed. Do not create custom packages during pilots.

## Tier A — Platform Access (Self-Serve)

Positioning:
> Operational clarity system for startup execution control.

Includes:
- SaaS platform access
- guided in-product onboarding script
- deterministic demo reset capability

Does not include:
- custom setup work
- custom workflows
- consulting/advisory sessions
- integrations

Hard constraints:
- No customization commitments
- No implementation labor beyond the guided script
- Scope is product access only

Ideal buyer profile:
- Founder or operator willing to self-execute onboarding
- Team with clear internal owner for rollout

---

## Tier B — Pilot Implementation (Guided Rollout)

Format:
- Fixed scope
- Time-boxed activation window (1-2 weeks)
- Ends at system activation

Includes:
- guided org setup session
- ownership mapping facilitation
- action system seeding
- policy structuring
- COO walkthrough execution
- handover completion checklist

Hard rule:
> This tier ends at activation and handover. It is not ongoing advisory.

Does not include:
- ongoing advisory retainer
- bespoke process design
- change management program
- custom feature delivery

---

## Non-negotiable boundaries (both tiers)

- No feature requests during pilot phase
- No roadmap commitments
- No custom development commitments
- No architecture defense conversations

Standard response when pressured outside scope:
> We are currently validating core operational clarity before expanding scope.

---

## Selection logic

Use Tier A when:
- client has internal operator capacity
- needs immediate product access

Use Tier B when:
- client needs structured rollout support
- ownership/action/policy setup needs facilitation

If neither fits:
- defer the opportunity until after pilot validation phase.
